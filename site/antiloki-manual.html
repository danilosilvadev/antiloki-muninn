the operator's manual

Run antiloki at full power.

antiloki is an autonomous AI development substrate that runs on your machine — your code, your data, and your record stay in files you own, and the AI itself can run locally. This manual covers every screen, the workflows that matter, the command line underneath, and the habits that separate using it from operating it.

01Getting started

Boot the two halves

# backend — the substrate engine + API (default 127.0.0.1:41931)
$ cd antiloki-BE && npm start
# frontend — the web app (Vite, default 5173, strict port)
$ cd antiloki-v3-FE && npm run dev

The backend binds loopback only — nothing is exposed to the network. Config lives in antiloki-BE/.env (host, port, flags); the frontend finds the backend via VITE_BE_URL.

First run: the workspace gate

Until a workspace is active, the app shows an onboarding gate instead of the shell: sign in (a Dev access button skips credentials for local use), then open an existing project folder or create a fresh one. A workspace is an external project antiloki manages — it gets its own private .antiloki/db set inside that project. Switch workspaces any time from the shell.

Connect a model (BYOK)

Go to Settings → BYOK, paste your OpenRouter key, and assign a model to each capability tier — low (the workhorse swarm), mid (judgment), high (the architect). Models come from the live provider catalog, never a stale list. The agents pick the tier deterministically; you pick what runs in it — including fully local models (Ollama and friends): bind a tier to one and even inference never leaves your machine. Every call is metered.

tip · the routing philosophy
Spend the expensive model only where judgment lives: reasoning legs run high, formatting/authoring legs run low. This “two-phase split” is the default — a well-configured low tier does most of the work for cents.

02The shell — always on screen

  • Command palette — search, jump to any screen or card, run allowed actions. The fastest way to move.
  • Side chat — the interviewer follows you to any screen; ask about what you're looking at.
  • Notifications drawer — pulses from the substrate: retention checks, driver warnings, weekly digests. An unread error-severity notification pauses the autonomous daemon — the system refuses to keep working past an unacknowledged failure. Dismissing it resumes work.
  • Health signals — the top bar surfaces backend reachability and driver-tick liveness (see Troubleshooting for “TICK LOOP DEAD”).

03Chat — the front door

/ · sidebar “Chat”

The interviewer

Talk to the substrate in plain language. The interviewer proposes structured actions (file a card, record a memory, change a status) and dispatches only after your “go” — conversation is never silently a mutation. Attach files with the paperclip; they're ingested into the knowledge base and ground future answers.

use it for
  • Filing work: describe it → it drafts a card with kind, size, and why → “go”.
  • Recording decisions and corrections as memories the system learns from.
  • Asking “what happened / what is X” — grounded in the actual databases.

04Board & Pipeline — mission control

/operations · sidebar “Board & Pipeline”

The board

Five columns: backlog / doing / pull requests / done / archived. The “pull requests” column is derived truth — QA-green work waiting for your merge, not a status anyone sets. Header shows real spend today. The board pans horizontally — grab any empty spot and drag.

key actions
  • ＋ propose card → opens the Features wizard (see §08).
  • Play / pause per card — the driver only works cards set to play. Pause is instant and always safe.
  • Click a card → its Waterfall. Badges show live phase, failures, and parent links.
pipeline sub-tab
  • The dispatch queue: what the driver will pick next and why each other card is blocked — the schedule made legible.

05Waterfall — one card's run, live

/operations/waterfall/<slug>

The phase ladder + everything about the run

Left: the ladder — screening, contract, blueprint design, design lint, the QA tribunal, build env, apply, build check, QA verify, seal. Green is done, red is where it stopped, and a red gate always says why and what it needs.

controls
  • Run / force tick — nudge the driver now instead of waiting for its loop.
  • Rewind — send the card back to an earlier phase. Your manual rewind is supreme (only already-shipped work blocks it); the system's automatic rewinds are budget-capped.
  • Reset — full restart of the card's run, with confirmation.
  • Auto-merge toggle — per card: merge instantly at QA-green (trusted lanes) or ship a PR and wait for you (default).
  • Stuck banner — when the run can't proceed, it tells you the blocking fact and pauses. Never “unblock and hope” — read it, fix the cause or rewind.
  • Clarify — when the planner needs an answer from you, the question appears inline; answering resumes the run.
tabs
  • contract (scope + acceptance) · blueprint / describe / diff / court (the design, its diff, and the tribunal's verdicts) · code · tests · build · prompts (every AI call with cost) · commits · cost · lock · drift · history — the card's provable history straight from the sealed chain.

06Pulls & Tests — the human gate

/operations/pulls · sidebar “Pulls & Tests”

The merge cockpit

Every QA-green change arrives here as a pull request with checks, AI risk notes, acceptance criteria, and blast radius. Merging is your act — the last gate is human on purpose. The tests side runs your real suite and shows exactly what's red, with baseline-aware honesty: failures that pre-exist your change are labeled as such, never blamed on it.

07Audit — the record, and the proof

/operations/audit

Three views: events · chain · prompts

events — the live activity log: every dispatched action and skill with actor, source, outcome, duration. Filter by text or source.

chain — the sealed record. This is the trust surface:

  • verify full chain — one button walks every archived segment plus the live chain, recomputes every hash, and checks sealed digests. Green banner = the entire history is intact; red names the exact segment and row.
  • sealed rows — the chain itself, filterable by entity type and id. Each row shows its print prefix.
  • archived segments — rotated history: rows, time span, compression; click for the segment's digest (a readable summary, stamped regenerable, NOT the record).
  • rotation — deliberately a copyable command, not a button: rotation requires the backend stopped (see workflow).

prompts — every AI call ever made: model, tier, cost, outcome; click for the full prompt and response.

The file mosaic sizes tiles by substance; click into the reading IDE; open the graph overlay; run blast radius on any node to see everything a change would touch. ↻ rebuild re-indexes in the background with per-phase progress; scope controls which folders/extensions count.

09Features — from intent to card

/features · also “＋ propose card” on the board

The control center + wizard

The feature inventory is derived deterministically from the atlas (entry points + data stories) — what the product actually has, not what a doc claims. The 3-step wizard turns an idea into a properly-shaped card: describe → refine scope/acceptance → file. Cards land in backlog, ready for the driver.

10Drift — reconcile reality with one click each

day-to-day: the waterfall's drift tab · cross-card feed at /drift (direct URL, off the sidebar)

The questionnaire pattern

When state diverges (a plan vs. what happened, a file vs. its record), drift becomes a question with concrete options. One option is always ★ recommended, derived from live state — and non-recommended choices say what they'd need. Clicks select; one re-run tick button applies everything and pushes the driver. You classify; the system executes.

In practice you meet drift inside the card that has it — the waterfall's drift tab — which is why the standalone cross-card feed lives off the sidebar. It remains useful as the "anything diverged anywhere?" sweep.

10bDesign — the self-generating design system

/operations/design · sidebar "Code & Design"

Your real UI, tokenized

Captures the actually rendered interface — not the design-tool ideal — extracts tokens (colors, type, spacing) and components from it, and flags drift between what was designed and what shipped. The design system writes itself from the product, so it can never lie about the product.

use it for
  • Auditing visual consistency after AI-built UI lands.
  • Handing agents a ground-truth token set so new screens match the old ones.
  • Catching the "shipped ≠ designed" gap before a user does.

10cTagging & Heimdall — your clicks make it smarter

/tagging · plus tag affordances on AI calls, failures, and verdicts

One click = one labeled example

Every tagging surface follows one grammar: the system surfaces items, you classify with one click — never forms, never free text. Each click lands as ground-truth evidence in Heimdall's ledger, the evidence-based casting router that sits on the single AI-call chokepoint and decides which model handles which kind of problem.

how the loop runs
  • Classify: every AI call is mapped to a problem category (build / design / judgment / language / reasoning) — deterministically for the substrate's own calls, AI-assisted only for free-form input.
  • Cast: Heimdall picks the worker from evidence at the right rung of the specificity ladder (template×model → category×model → family×model → global → default) — the first level with statistical footing wins, so new templates inherit their category's wisdom.
  • Record: outcomes flow back — machine gates (parse, build, ship) weigh more than AI verdicts, which weigh more than benchmarks. Your tags are the highest-value signal of all.
  • Guard rails: flip-flop detection catches unstable classifications; capability ceilings and quarantine are hard filters a good score can never buy past; and attribution is never swallowed — every surface shows HEIMDALL → <worker>.

The weekly digest in your notifications reports the loop's pulse: evidence added, tags added, categories now speaking with their own voice.

tip · tag the bad ones especially
"Which model wrote this garbage?" is the exact instinct the ledger runs on. A minute of tagging failures teaches the router more than an hour of anything else.

13Workflow — idea to merged PR

  1. File it: board → ＋ propose → wizard (or just tell the chat). A scored card lands in backlog.
  2. Set it to play. The driver picks it up on its loop — watch the Waterfall.
  3. Answer if called. Design failed its tribunal twice? It escalates to you with the verdict. Clarify questions appear inline.
  4. Merge. QA-green work waits in Pulls with risk + blast radius. Your click ships it.
  5. Trust, then verify. Waterfall → history shows the card's sealed rows; Audit → chain → verify proves the whole record.

14Workflow — a stuck or failing card

  1. Read the stuck banner — it names the blocking fact.
  2. Check the failing tab (court for design verdicts, tests·build for gates, prompts for the AI leg that misfired).
  3. Fix the cause (often: answer clarify, adjust the contract, or accept a drift recommendation) — then rewind to the right phase and let it re-run.
  4. If it keeps failing the same way: pause it. A card that fails twice identically is telling you about the plan, not the executor.
rule of thumb
Never chase a red gate by re-running it unchanged. The gates are deterministic — same input, same red. Change something, then rewind.

15Workflow — prove the record (10 seconds)

  1. Operations → Audit → chain → verify full chain.
  2. Green: every row of history, including compressed archives, re-hashed and intact.
  3. Red: it names the segment and row. That's evidence of tampering or corruption — treat it as an incident, restore the genuine archive from your backup/second copy, verify again.
# the same proof from a terminal, offline:
$ ANTILOKI_DB=.antiloki/db/state.sqlite node bin/antiloki.mjs system.timeline.verify_full '{}'
{"valid":true,"total":1489,"archived_segments":1,...}

16Workflow — rotate the timeline (when nudged)

The chain grows slowly (~25 rows/day typical). A daily pulse watches it; when it crosses thresholds you'll get a notification suggesting rotation. Rotation archives the live segment (gzip + sealed digest) and opens a fresh one — chain continuity is cryptographic across the file boundary.

  1. Stop the backend (rotation needs single-writer access — the screen will remind you).
  2. Dry-run, then run:
$ ANTILOKI_DB=.antiloki/db/state.sqlite node bin/antiloki.mjs system.timeline.rotate '{"dry_run":true}'
$ ANTILOKI_DB=.antiloki/db/state.sqlite node bin/antiloki.mjs system.timeline.rotate '{"copy_dir":"D:/backups/antiloki-archives"}'
  1. Restart the backend. Verify the chain (workflow 15) — it must be green across the new boundary.
tip · the second copy
Pass copy_dir to write archive copies somewhere safe. A sealed-but-deleted archive is detectable forever but recoverable only if a copy exists.

17Workflow — control spend

  • Watch the board header (today) and /cost (everything, per call).
  • Put a cheap/local model on low — it does most of the volume. Reserve high for the architect legs.
  • Per-card cost lives in the waterfall's cost tab; per-call detail one click deeper.
  • Tests can never spend your money: provider calls from test contexts are refused by a guarded chokepoint (with its own canary).

17bEnvironments & isolation

antiloki never lets an agent build in your working tree, and it never lets configuration hide.

  • Per-card build isolation. Every card's build happens in its own git worktree (automatic when the workspace is a git repo) or a copy-on-write sandbox — your checkout is untouched until a pull request merges. The sealed record stamps exec_env on every action, so "where did this run" is part of the proof.
  • Per-workspace Docker isolation. Managed projects can run their runtime inside a dedicated container — one container per workspace, with the host reserved for the substrate itself. The lifecycle is managed for you; the default stays inert until you opt a workspace in.
  • The env-flag registry. Every environment flag the system reads is a registered row with a category, a danger tier (safe · caution · enforcement-defeating), where it's read, and its default — rendered to docs/env-flags.md and census-checked by tests so no undocumented switch can exist. The backend warns loudly at boot if any enforcement-defeating flag is set: weakening the guarantees is allowed on your own machine, but never silently.
tip · reading the registry
Open docs/env-flags.md (also viewable in-app at /docs) before touching .env — every flag's blast radius is documented next to its name.

18The CLI — the same substrate, no browser

# discovery — the substrate is queryable, not narrative
$ node bin/antiloki.mjs system.law.find '{"query":"audit"}'
$ node bin/antiloki.mjs system.action.find '{"query":"timeline"}'
$ node bin/antiloki.mjs system.consistency.report '{}'     # 18 claim-vs-reality dimensions
$ node bin/antiloki.mjs board-keeper.log.board '{}'        # the board, as text

# migrations (single one by name; never blanket-run on an old DB)
$ ANTILOKI_DB=.antiloki/db/state.sqlite node bin/antiloki.mjs system.migrate.run '{"only":"<name>","dry_run":true}'

Targeting: ANTILOKI_DB pins which substrate you're talking to (relative .antiloki/db/state.sqlite from a project root = that project); all nine databases resolve as siblings of it — one rule, no split-brain. Mutations from the CLI are gated: unsigned mutating dispatches are refused by law; reads and the bootstrap allowlist work plainly. Privileged overrides exist (an ed25519-signed “god” challenge) for interactive human use — if you need it, you'll know, and the chain records the exception.

19What the system will refuse — by design

these are features
  • Editing or deleting audit rows — refused by database triggers, for everyone.
  • Unsigned mutations from scripts/CLI — the dispatch gate demands driver HMAC, a signed override, or an allowlisted read.
  • Faking protocol state — hand-written audit rows, forged QA passes: hard-law violations; the chain can't distinguish forged from real, so writing it any way but through the gate is forbidden absolutely.
  • Rotating the timeline under a live backend — the skill checks its preconditions and refuses with the healing step.
  • Committing secrets — pre-commit refuses; the auth directory is ignored by construction.
  • Destructive operations without a fresh yes — always-ask does not remember yesterday's consent.

The full law book renders into the repository's AGENTS.md automatically — 37 active laws with rationale and provenance. Query them: system.law.find.

21Max power — the operator habits

  • Trust the pause. The system stopping is it working. Read the banner before touching anything.
  • File everything through the wizard. Well-shaped cards (kind, size, why, acceptance) are the difference between a driver that ships and one that asks.
  • Let low-tier do the volume. Your high-tier model is an architect, not a typist — the cost tab will show you when you've got the split right.
  • Verify weekly, rotate rarely, copy always. Chain-verify is 10 seconds; make it a habit. Rotation only when nudged — always with copy_dir.
  • Answer drift with the star. The ★ recommendation is computed from live state — deviate when you know better, but know you're deviating.
  • Read the history tab after anything surprising. “What provably happened” beats “what I remember happening”, especially about your own actions.
  • Use dry-run everywhere it exists (rotation, migrations, retention). The system offers it because it wants to show you the plan first.
  • Query, don't guess: system.law.find, system.action.find, the consistency report. The substrate answers precisely; the habit compounds.
